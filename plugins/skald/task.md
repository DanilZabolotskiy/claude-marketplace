## Stories

### Story 22: Прием платежей через CryptoCloud

**Цель:** Добавить криптовалюту как второй способ оплаты подписки STANDARD наряду с Telegram Stars. Пользователь выбирает период (1/3/6/12 мес), затем на экране выбора способа оплаты нажимает кнопку «Криптовалюта» — бот формирует инвойс через API CryptoCloud и показывает сообщение с кнопкой-ссылкой на платёжную страницу. Идентификатор инвойса сохраняется и привязывается к чату. После подтверждения платежа CryptoCloud шлёт постбек на наш эндпоинт `/payment/cryptocloud`, по которому мы находим сохранённый инвойс и активируем подписку нужному чату. Существующий путь Telegram Stars по поведению не меняется.

**Документация провайдера:** https://docs.cryptocloud.plus/en

**Ключи доступа (уже добавлены в `.env`):**
- `CRYPTO_CLOUD_SHOP_ID` — идентификатор магазина
- `CRYPTO_CLOUD_API_KEY` — API-токен для заголовка `Authorization: Token …`
- `CRYPTO_CLOUD_SECRET_KEY` — секрет для проверки JWT-подписи постбека (HS256)

**Соглашения по именованию:**
- Все доменные и инфраструктурные типы, относящиеся к провайдеру, именуются с префиксом `CryptoCloud*` (например `CryptoCloudInvoice`, `CryptoCloudPostbackClaims`, `CryptoCloudInvoiceRecord`, `CryptoCloudInvoiceStatus`, `CryptoCloudPaymentPort`, `CryptoCloudInvoiceRepositoryPort`, `CryptoCloudAdapter`, `CryptoCloudClient`, `CryptoCloudProperties`, `CryptoCloudAutoConfiguration`, `CryptoCloudWebhookController`).
- Это позволяет в будущем добавить других крипто-провайдеров без конфликта имён.

#### Итерация 22.1: Вынести активацию token pack в application use-case

- **Проблема:** Логика зачисления token pack (инкремент бонусного лимита подписки, создание `QuestionQuota`) сейчас живёт в `BillingHandler` внутри инфраструктурного модуля бота. Переиспользовать её из вебхука CryptoCloud нельзя — нужен общий, независимый от платёжного канала use-case. Это зеркало итерации 22.1 для подписок.
- **Что делаем:**
  - Создаём новый use-case в доменном/application-слое, принимающий идентификатор чата и количество сообщений.
  - В use-case переносим текущую логику зачисления: инкремент бонусного лимита и создание `QuestionQuota` с источником token pack.
  - Переводим `BillingHandler` при успешной оплате звёздами на делегирование в новый use-case.
  - За `BillingHandler` оставляем только парсинг Telegram-payload и формирование локализованного текста подтверждения.

**Результат:** Бизнес-логика покупки token pack готова к переиспользованию из двух платёжных каналов (Stars и CryptoCloud); регресс по Stars-пути отсутствует.

#### Итерация 22.2: Расширение модели и таблицы `crypto_invoices` на два типа покупок

- **Проблема:** Текущая модель инвойса поддерживает только подписку — обязательное поле «число месяцев». Для token pack нужно хранить количество сообщений, а чтобы различать типы — дискриминатор.
- **Что делаем:**
  - Вводим enum с типами покупки (подписка / token pack).
  - Расширяем доменную модель `CryptoCloudInvoiceRecord` дискриминатором и опциональным полем под количество сообщений; поле «число месяцев» делаем опциональным.
  - Добавляем в модель доменный инвариант: ровно одно из двух полей заполнено в зависимости от типа.
  - Миграция БД: добавляем колонки дискриминатора и количества сообщений, делаем «число месяцев» nullable, проставляем всем существующим записям тип «подписка».
  - Обновляем адаптер репозитория и маппинг под новые поля.

**Результат:** Одна таблица `crypto_invoices` поддерживает оба типа покупок; существующие подписочные записи продолжают работать без изменений.

#### Итерация 22.3: Бот — кнопка «Криптовалюта» в выборе способа оплаты token pack

- **Проблема:** На экране выбора способа оплаты доп. лимита показана только кнопка Telegram Stars. Пользователь не может заплатить криптовалютой.
- **Что делаем:**
  - В меню выбора способа оплаты token pack добавляем вторую кнопку «Криптовалюта» — с тем же эмодзи, локализованным текстом и ценой в USD, что и на экране подписки; переиспользуем существующие ключи локализации.
  - Вводим новый callback-префикс для token-pack-crypto и мапу USD-цен пакетов.
  - В существующий `CryptoCloudBillingHandler` добавляем метод обработки этого callback: создание инвойса через порт CryptoCloud, сохранение записи с типом «token pack», возврат сообщения с URL-кнопкой на платёжную страницу.
  - При ошибке провайдера возвращаем локализованный fallback, запись в БД не создаём.
  - В `TelegramBotService` маршрутизируем новый callback в этот метод.
  - Существующий метод обработки crypto-подписки по поведению не меняем — только проставляем тип «подписка» при создании записи.

**Результат:** Пользователь на экране оплаты доп. лимита видит кнопку «Криптовалюта» с правильной ценой; по нажатию получает сообщение с ссылкой на CryptoCloud; инвойс сохранён в БД с привязкой к чату и типом «token pack».

#### Итерация 22.4: Активация token pack по постбеку CryptoCloud

- **Проблема:** Текущий сервис обработки постбека всегда активирует подписку — он ничего не знает про token pack. Для записей с типом «token pack» поведение должно быть другим: зачислить доп. лимит, а не подписку.
- **Что делаем:**
  - В сервисе обработки постбека добавляем ветвление по типу покупки записи.
  - Для типа «подписка» — оставляем текущее поведение (активация use-case'ом подписки, сообщение `PAYMENT_SUCCESS`).
  - Для типа «token pack» — вызываем новый use-case из 22.8 с количеством сообщений из записи и отправляем пользователю сообщение `PAYMENT_TOKENS_SUCCESS` (тот же, что в Stars-пути).
  - Идемпотентность по статусу `PAID`, пометку инвойса как оплаченного и отправку сообщения оставляем общими для обеих веток.
  - Обрабатываем случай, когда на момент подтверждения у чата нет активной подписки (token pack нельзя начислить): логируем, инвойс оставляем в `CREATED`, отвечаем 200.

**Результат:** Вебхук `/payment/cryptocloud` корректно обрабатывает оба типа покупок; пользователь, оплативший доп. лимит криптовалютой, получает то же подтверждающее сообщение, что и при оплате звёздами; регресс по crypto-флоу подписок отсутствует. После выполнения отметить итерации 22.8–22.11 в [iteration.md](iteration.md).
